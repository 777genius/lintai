## 1. Что мы строим (одним абзацем)

**lintai** — офлайн-first **линтер для agent supply chain** в репозитории: agent skills, MCP-конфигурации, хуки и связанные артефакты (включая Cursor Plugins). Один **нативный бинарь**, детерминированные правила по умолчанию, **SARIF** и нормальные exit codes для CI. Позиционирование: **«Ruff/Biome для контекста агента»**, не «ещё одна облачная security platform». Более широкие ambitions вроде deeper engines или расширенного auto-fix относятся к post-beta развитию, а не к базовому обещанию продукта.

## 2. Продукт v0.1 (минимум, который уже «продукт»)

| Входит в v0.1 | Не входит в v0.1 (позже) |
|-----------------|---------------------------|
| `lintai scan` по репозиторию | Полноценный LSP |
| Форматы: `SKILL.md`, `CLAUDE.md`, `.mdc` / `.cursorrules`, `mcp.json`, **Cursor Plugins** | Все 14+ платформ сразу |
| Стабильные **rule id**, SARIF + текст/JSON | Pre-install по всем регистрам |
| Сборка на **всех целевых таргетах** с первого релиза (вкл. Windows, Linux musl) | Тяжёлый YARA-enabled бинарь как единственный артефакт |
| Скорость и offline как нефункциональные требования | Облачная аналитика, аккаунты, телеметрия по умолчанию |

**Одна строка для README:**  
*Локальный линтер для skills + MCP + плагинов в CI: SARIF, без облака по умолчанию.*

## 3. Стратегия кода: не велосипед, но свой каркас

| Принцип | Смысл |
|---------|--------|
| **Свой каркас** | Движок правил, модель finding’ов, адаптеры форматов, CLI, SARIF pipeline, fix pipeline, публичный API (`lintai-api`). |
| **Чужое — данные и проверенные паттерны** | Regex/YARA/таксономии из OSS; адаптация под единый движок lintai. |
| **Библиотеки Rust — стандарт экосистемы** | Парсинг markdown/yaml/json, YARA-X, CLI UX — через crates, не самописные парсеры с нуля. |
| **cc-audit** | **Селективный порт** правил, тестов, эвристик, CWE-маппинга — не обязательный monolithic fork всего репозитория. |
| **Лицензии** | Каждый перенос — в [CREDITS.md](CREDITS.md); соблюдение MIT/Apache/CC BY и т.д. |

## 4. Дифференциация (почему не «ещё один сканер»)

1. **Offline / air-gap по умолчанию** — контраст с облачными scanner’ами.  
2. **Один бинарь, предсказуемая стоимость CI** — контраст с Python + LLM + API.  
3. **Skills + MCP + plugins в одном UX** — не раздельные Cisco skill vs mcp только.  
4. **SARIF как контракт** — встраивание в существующий DevSecOps без новой консоли.  
5. **Remediation (по фазам)** — сначала узкий безопасный набор, затем осторожное расширение по evidence.

Реестры (skills.sh + аудиты) закрывают **публичный** каталог; lintai усиливает **приватный монорепо, pre-push, кастомные правила, air-gap**.

---

## 5. Конкурентный контекст (кратко)

Ниша **фрагментирована**: полного OSS-аналога «всё в одном» под наш клин нет; **cc-audit** ближе всего по функциям; **skillscan-security** и др. — частичные пересечения. Окно **сужается** из-за Snyk, Cisco, Vercel/registry — усиливает ставку на **скорость выхода v0.1** и чёткое позиционирование.

Актуальные цифры и список игроков: [COMPETITOR_ANALYSIS.md](../../research/COMPETITOR_ANALYSIS.md).

## 6. Два стратегических режима (на будущее)

| Режим | Когда |
|--------|--------|
| **A — дифференциаторы** | Упор на air-gap, скорость, SARIF, narrow remediation и кастомные правила — то, что облачным вендорам экономически тяжело сделать дефолтом. |
| **B — complementary** | Явно «рядом с Snyk» в корпорациях: GitHub Action, pre-commit, не позиция «замените платформу». |

Выбор канала GTM не меняет ядро продукта из §1–2.
