# Ресёрч: автоматическая документация для rules через VitePress (с акцентом на свежесть и Rust)

Дата фиксации: 2026-03-29  
Контекст: `/Users/belief/dev/projects/agent_skills`

## 1) Вопрос и цель

Проверено: можно ли строить автогенерацию документации для наших rules через `vitepress` (желательно v2), так чтобы docs были всегда свежими, и как это лучше совмещать с ручными markdown-правилами.

Дополнительно:
- насколько зрелой/рискованной выглядит VitePress 2 на сейчас;
- есть ли полезные Rust-интеграции;
- есть ли более подходящая альтернатива или гибрид.

## 2) Что с VitePress 2 по состоянию на 2026-03-29

- Стабильная версия в npm на тот момент — `1.6.4`, по данным npm.
- Линейка 2.0 идёт как `2.0.0-alpha.*` (фиксируем `2.0.0-alpha.17` как релизная точка по публичным релизам VueJS).
- Официальный сайт `vitepress.dev` в это время документирует больше v2/next-ветку, а не только стабильную 1.x.
- Вывод: v2-ветка очень “свежая” по API/фичам, но это по сути alpha-платформа с рисками для production.

### Что из коробки подходит под автоматизацию в VitePress

Потенциально релевантные primitives:
- `srcDir`, `rewrites`, `themeConfig` в site config.
- `async config` + JS/TS-конфиг.
- `createContentLoader`, `defineLoader`/`defineRoutes` для загрузки данных и динамического роутинга на build-time.
- build-хуки (`transformPageData`, `transformHead`, `transformHtml`, `buildEnd`, `postRender`) для автоматизации.
- Vue/markdown-рендер в md файлах, frontmatter, data-driven навигация, i18n, shiki и т.д.
- Встроенного автоматического авто-sidebar “из коробки” нет; его обычно делают либо скриптом в пайплайне, либо плагинами/внешними генераторами.

### Ограничения и риски v2

- `v2` сейчас alpha: риск нестабильности API и поведения в edge-кейсах.
- Кастомизация default theme через `alias` может быть менее стабильной между релизами.
- Для больших dynamic-сайтов (много autogenerated страниц) выше требования к сборке и контролю памяти.
- Для проекта с высоким уровнем стабильности production иногда безопаснее стартовать с v1.6.4 и мигрировать позже.

## 3) Rust интеграции: что реально рабочее

### A) Rust-first вариант (рекомендованный для долгой поддержки)
1. Генерировать `rustdoc` JSON (часто через nightly/unstable output-format).
2. Прогонять через `cargo xtask`/скрипт и строить:
   - `api-manifest.json`
   - `api-sidebar.json`
   - `api-search.json`
   - опционально `docs/api/**/*.md` для human-readable частей.
3. VitePress читает артефакты через data loaders/async config и строит роуты/навигацию на сборке.

Плюсы:
- единый структурный источник истины;
- лучше масштабируется для freshness и ссылочной целостности;
- проще делать детект stale/дрейфа.

Минусы:
- `rustdoc json` в этой модели опирается на unstable-настройки toolchain;
- повышенная сложность оркестрации (xtask/CI).

### B) rustdoc/json -> markdown конвертеры
Инструменты типа `rustdoc-md`, `rustdoc-markdown` дают удобный markdown-выход, но хуже подходят как стабильный машино-ориентированный источник для сложной навигации и индексов.

Подход:
- годен для быстрого старта и лёгкой ревью-дифф в git,
- слабее для полноценного “single source of truth” и строгого rebuild-дисциплины.

### C) docs.rs JSON + VitePress
Можно использовать опубликованные артефакты docs.rs для некоторых публичных crate.

Плюс:
- меньше локальной генерации.

Минус:
- не всегда покрывает private/internal API и внутреннюю версионность проекта.

### D) mdBook как доп. слой
Рекомендуем только для narrative guide/обучающих частей, если нужны include/preprocessor/тестирование docs-snippets.

Плюс:
- зрелый Rust stack для markdown-книг.

Минус:
- не самый сильный backend для API-структур и связки с единой навигацией в одном портале.

## 4) Сочетание autogenerated + ручных MD

Практика с лучшей управляемостью:

- Делим контент на:
  - `docs/rules/` (ручные md: политика/руководства/FAQ).
  - `docs/generated/` (машинная генерация).
- Генерация запускается до `vitepress build`.
- Генератор пишет:
  - `.generated/nav-manifest.json` (источник для sidebar).
  - `.generated/sources.json` (привязка rule-id -> pages).
  - `.generated/version.json` (hashes входов для freshness checks).
- На CI:
  - `docs:generate` → `docs:lint-generated` (отсутствие stale-файлов/битые include).
  - `vitepress build` → `link check` → деплой.
- Формат `frontmatter` для ручных страниц фиксируется единым schema (например title/order/tags/published/deprecation/apiRefs).
- Для generated секций лучше запрещать ручные правки того же пути.

## 5) Что делать с sidebar и навигацией

Для автогенерации sidebar:

- базовый путь: генерация `.json`/`.ts` манифеста + async-конфиг в `.vitepress/config.ts`;
- альтернативно: плагинный слой из экосистемы (`vitepress-sidebar`, `nolebase`) на базе структуры файлов;
- сильный вариант: единый manifest-генератор с контролем сортировок, exclusions, grouping.

## 6) Рекомендуемый pipeline (практичный)

- `pnpm dlx`/`bunx`/`node` скрипт собирает generated артефакты в `docs/generated`.
- Rust-сегмент:
  - `cargo xtask gen-docs` (или shell-скрипт) для Rust API JSON + нормализации;
  - `cargo +nightly rustdoc --output-format json` (при необходимости);
  - преобразование в stable JSON-модель.
- VitePress этап:
  - `vitepress build`/`vitepress preview`.
- Проверки:
  - сверка drift (`git diff --exit-code` после генерации);
  - broken links;
  - schema validation для frontmatter/manual index.
- Релиз:
  - кеширование по хэшам источников (manual+generated+config+templates),
  - инвалидация только изменяемых сегментов через CI path rules.

## 8) Практическая рекомендация под текущий кейс

Если коротко: идея использовать VitePress имеет смысл, **но не как единственный источник истины**.  
Лучше строить архитектуру:

1. `manual rules md` + `generated api/metadata` = два слоя;
2. генераторы в CI (Rust: rustdoc JSON → нормализатор → manifest);
3. VitePress читает данные build-time и формирует сайт.

Это даёт свежесть и контроль.

При этом я бы стартовал с:
- v1.6.4 для production-гарантий,
- v2-compatible конфигом/паттернами,
- миграцией на v2 alpha после закрепления пайплайна автогенерации.

Если цель — “вынуть всё из эксперимента как можно быстрее”, можно идти сразу в v2, но с жесткими smoke-тестами сборки/deploy и запретом на глубокую зависимость от внутренних частей темы.

## 8.5) Доп. раунд: 3 варианта интеграции (по 3 новым агентам)

Ниже — отдельный блок по интеграции, который пришёл из дополнительного исследования на 3x саб-агента.

### Вариант 1 — GitOps markdown (рекомендуемый старт для team-review)

Идея: генерировать markdown в репозиторий в отдельные папки, коммитить diff и контролировать его дрейф.

- `docs/rules/**` — ручные правила и policy.
- `docs/api/typedoc/**` — автогенерируемые TS/JS API pages.
- `docs/api/vendor/**` — автогенерируемый openapi/vendor блок.
- `docs/generated/*.json` — манифесты `sidebar`, `sources`, `fingerprint`.

Пайплайн:
- запуск генераторов (typedoc + openapi-пайплайн),
- запись fingerprint-файла `docs/.meta/docs-fingerprints.json`,
- `git diff --exit-code docs/` как hard check на stale.

- Уверенность: `9/10`
- Надёжность: `9/10`
- Скорость внедрения: `6/10`

### Вариант 2 — CI-only generated (`.generated`, не коммитим)

Идея: не хранить generated MD в git; генерировать только во время build (в tmp/` .generated`), включать в source для сборки.

- меньше шума в git,
- быстрее локально и чище PRы,
- свежесть контролируется через manifest/hash в CI (не через diff репозитория).

- Уверенность: `8/10`
- Надёжность: `7/10`
- Скорость внедрения: `8/10`

### Вариант 3 — Data-driven `json -> template`

Идея: минимум markdown-генерации, больше структурных `.json` артефактов и рендера через VitePress data loaders (`createContentLoader`, `defineLoader`, `.data.ts`) и шаблонных страниц.

- гибко для API и vendor, лучше контроль структуры,
- меньше дублей между generated файлами,
- более сложная конфигурация и отладка.

- Уверенность: `6/10`
- Надёжность: `6/10`
- Скорость внедрения: `7/10`

### Рекомендованный вариант из этого блока

Для нашего кейса с правилами + автогенерацией лучше всего подходит Вариант 1:
- он прозрачен для ревью,
- быстро ловит stale-фактические расхождения через diff,
- проще объяснить ответственность между командой docs и автогенератором.

## 9) Edge cases и как лучше сделать

Ниже — практические edge cases, которые обычно дают инциденты в production docs pipelines, и меры с минимальной сложностью внедрения.

1. stale docs после merge
   - Симптом: код обновился, docs остались старые.
   - Причина: инкрементальный режим не охватывает все зависимости.
   - Риск: высокий.
   - Лучшая правка: зависимостный manifest + обязательный сравниваемый `source_hash`/`generated_hash`.

2. частичный сбой генерации
   - Симптом: часть новых страниц есть, часть нет, пайплайн зелёный.
   - Причина: генератор пишет постепенно без fail-fast.
   - Риск: высокий.
   - Лучшая правка: atomic-пайплайн в temp dir + финальная проверка count + `exit 1` при неполном наборе.

3. удаление правил и «висячие» страницы
   - Симптом: в меню ещё видны несуществующие rule pages.
   - Причина: deleted rule не удаляет generated-страницы.
   - Риск: средний/высокий.
   - Лучшая правка: inventory-мэп `source->generated`, очистка orphan, fail если orphan вне allowlist.

4. ссылочный дрейф (slug/redirect)
   - Симптом: 404 после рефакторинга id/slug.
   - Причина: ручные ссылки не синхронизированы с изменёнными slug.
   - Риск: высокий.
   - Лучшая правка: стабильный rule-id + `old_slug->new_slug` + обязательный `brokenLinks: error`.

5. изменения toolchain/lockfile
   - Симптом: локально проходит, в CI падает или контент разъехался.
   - Причина: плавающие зависимости.
   - Риск: высокий.
   - Лучшая правка: зафиксировать `node`/менеджер/пакеты (`packageManager`, lockfile), run через immutable installs.

6. несоответствие dev и CI
   - Симптом: локально page работает, в CI падает.
   - Причина: разный env, flags, кэш, не повторяемые команды.
   - Риск: средний.
   - Лучшая правка: единый `docs:build` скрипт, одинаковые env, run в docker-образе как у CI.

7. i18n fallback и локали
   - Симптом: страницы пропадают из одной локали или показываются fallback-языком.
   - Причина: невалидный locale manifest.
   - Риск: средний.
   - Лучшая правка: валидация locale-schema и строгая политика fallback только для не-критичных разделов.

8. route slug collision
   - Симптом: разные правила схлопываются в один URL.
   - Причина: нормализация slug и case/cyrillic collisions.
   - Риск: высокий.
   - Лучшая правка: детект дублирующихся canonical URL до публикации, детерминированное правило суффиксов.

9. inconsistency search index
   - Симптом: поиск ведет на несуществующие страницы.
   - Причина: индекс строился из другого набора файлов.
   - Риск: средний.
   - Лучшая правка: индекс только после финальной сборки + проверка каждого hit-path через фактический route map.

10. broken frontmatter
   - Симптом: страницы не попадают в nav/search или рендерятся некорректно.
   - Причина: YAML ошибки/nullable поля.
   - Риск: средний.
   - Лучшая правка: JSON Schema / Zod для frontmatter + pre-commit и CI-валидатор.

11. гонки deploy/параллельность
   - Симптом: один успешный билд затирает другой.
   - Причина: параллельные публикации одного ref.
   - Риск: высокий.
   - Лучшая правка: `concurrency` в GitHub Actions и серийная публикация с atomic switch.

12. partial checkout / shallow clone
   - Симптом: сбрасывается история для diff-based checks.
   - Причина: `fetch-depth` слишком мал.
   - Риск: средний.
   - Лучшая правка: `fetch-depth: 0`, `fetch-tags: true`, `submodules: recursive`.

13. case-sensitivity и symlink
   - Симптом: локально видно, в Linux падает.
   - Причина: файловая система/регистры путей.
   - Риск: средний.
   - Лучшая правка: запрет case-collisions и запрет symlink в docs (или их нормализация до копирования).

14. security: untrusted content в generated md
   - Симптом: XSS или HTML-инъекция в рендере docs.
   - Причина: генератор пропускает raw HTML или unsafe markdown из входных данных.
   - Риск: высокий.
   - Лучшая правка: строгий sanitizer на входе, запрет raw HTML в md, allowlist тегов и атрибутов.

15. publish cache poisoning / stale CDN
   - Симптом: после деплоя видны старые страницы или смесь версий.
   - Причина: CDN не инвалитится полностью, совпадающие пути между версиями.
   - Риск: высокий.
   - Лучшая правка: версионированные пути (`/vX/`), cache-busting headers, atomic publish и глобальная invalidation.

16. versioning mismatch (rules vN vs code vM)
   - Симптом: docs показывают правила из другой версии кода.
   - Причина: генерация и release тег не синхронизированы.
   - Риск: высокий.
   - Лучшая правка: single source tag (git tag/commit), build docs только на тегах, проверка `version.json` на соответствие.

17. масштабирование build (N>10k страниц)
   - Симптом: build time и memory деградируют, падения VitePress.
   - Причина: слишком много страниц без шардирования.
   - Риск: средний/высокий.
   - Лучшая правка: батч-генерация, разделение на пакеты/версии, отключение лишних transforms, use cache.

18. redirects loop / redirect chain explosion
   - Симптом: пользователь или crawler застревает на цепочке редиректов.
   - Причина: автоматические редиректы строятся без проверки циклов.
   - Риск: средний/высокий.
   - Лучшая правка: детект циклов и ограничение длины цепочки, unit-check для `redirects.json`.

19. link integrity: сломанные якоря внутри страниц
   - Симптом: ссылка ведёт на страницу, но якорь 404/не найден.
   - Причина: auto-slug/heading меняется при генерации.
   - Риск: средний.
   - Лучшая правка: проверка `route + hash` на реальном HTML после build, а не только по файлам.

20. race condition: генератор vs сборка
   - Симптом: build читает частично обновлённые файлы.
   - Причина: генерация и build идут параллельно в одном workspace.
   - Риск: высокий.
   - Лучшая правка: писать в temp dir и только потом atomic move, либо разделить jobs.

21. stale build artifacts / кеши
   - Симптом: новая генерация не отражается в build output.
   - Причина: кэш Vite/VitePress/CI не инвалитится при изменениях источников.
   - Риск: средний.
   - Лучшая правка: кэш-ключи от `source_hash` и чистка output перед сборкой.

22. permissions в CI (private repos/artifacts)
   - Симптом: генерация не видит входные правила/схемы.
   - Причина: токен без нужных прав, forked PR без secrets.
   - Риск: средний/высокий.
   - Лучшая правка: разделять trusted/untrusted workflows, read-only токены, preflight проверка доступа.

23. secret leakage в docs
   - Симптом: секреты случайно попадают в опубликованные страницы.
   - Причина: генератор читает env/лог или не фильтрует конфиги.
   - Риск: высокий.
   - Лучшая правка: строгий allowlist данных, redaction, lint на `secret patterns` перед publish.

24. локализация и redirects пересекаются
   - Симптом: редирект перекидывает на неверную локаль.
   - Причина: редиректы генерируются без привязки `locale`.
   - Риск: средний.
   - Лучшая правка: locale-aware redirects и тест на соответствие `locale -> locale`.

25. SSR-only crash в markdown/Vue компонентах
   - Симптом: билд падает только в CI/SSR, локально dev всё ок.
   - Причина: использование `window/document` в md-компонентах без guard.
   - Риск: 7/10.
   - Лучшая правка: `if (typeof window !== "undefined")`, вынести DOM-доступ в `onMounted`, SSR-safe компоненты.

26. hydration mismatch на generated-страницах
   - Симптом: в проде предупреждения о mismatch, часть UI “прыгает”.
   - Причина: нереплицируемый рендер (рандом, время, порядок данных).
   - Риск: 6/10.
   - Лучшая правка: детерминированный рендер, фиксированный sort, исключить `Date.now()`/`Math.random()` из SSR.

27. диагностика сборки скрыта (ложно-зелёный пайплайн)
   - Симптом: сборка зелёная, но часть страниц отсутствует.
   - Причина: warnings не фейлят build, лог не сохраняется.
   - Риск: 7/10.
   - Лучшая правка: `--strict`, fail-on-warning для docs, сохранение артефактов логов.

28. fork PR без секретов ломает генерацию
   - Симптом: docs генерятся частично или не генерятся вовсе на PR из форка.
   - Причина: секреты недоступны, private API/репозитории недоступны.
   - Риск: 6/10.
   - Лучшая правка: split pipeline (trusted vs untrusted), мок-данные или пропуск генерации с явной меткой.

29. зависимые репозитории / submodules меняют API без синка
   - Симптом: docs не соответствуют фактическому API.
   - Причина: docs берут данные из submodule/dep с плавающей версией.
   - Риск: 8/10.
   - Лучшая правка: pin на tag/commit, проверка совместимости версий на входе.

30. миграция версий ломает SEO canonical/hreflang
   - Симптом: поисковики индексируют дубликаты, падают позиции.
   - Причина: неверные `canonical`/`hreflang` после изменения структуры.
   - Риск: 5/10.
   - Лучшая правка: единая canonical-логика, тесты sitemap + hreflang.

31. сломанный sitemap для generated routes
   - Симптом: новые страницы не индексируются.
   - Причина: sitemap строится до генерации или не включает `.generated`.
   - Риск: 5/10.
   - Лучшая правка: строить sitemap после финального build, валидировать count маршрутов.

32. кэш-слои дают “временной разнобой”
   - Симптом: часть пользователей видит старую навигацию, часть — новую.
   - Причина: разные TTL на CDN, поисковом индексе, сервис-воркере.
   - Риск: 6/10.
   - Лучшая правка: единая версия release-id, сквозной cache-bust, синхронная инвалидация.

33. лимиты пакетов/реестров (npm/pnpm rate limit)
   - Симптом: периодические падения установок зависимостей в CI.
   - Причина: rate limit или недоступность реестра.
   - Риск: 4/10.
   - Лучшая правка: registry cache/proxy, retries, локальный mirror.

34. локализация и SEO-конфликты slugs
   - Симптом: одна локаль перетирает другую в индексе.
   - Причина: одинаковые slug без `locale`-префикса.
   - Риск: 6/10.
   - Лучшая правка: обязательный locale-prefix в URL, проверка уникальности `route` на уровне билда.

35. legal/compliance: нет обязательных legal-страниц в релизе
   - Симптом: отсутствуют Terms/Privacy/License или устаревшие версии.
   - Причина: генерация не проверяет обязательные документы на присутствие/актуальность.
   - Вероятность: 4/10.
   - Ущерб: 9/10.
   - Лучшая правка: обязательный compliance-check с allowlist страниц и версиями, блокировка релиза при отсутствии.

36. git tags/branching: docs не соответствуют релизному тегу
   - Симптом: опубликованы страницы от `main`, а релизная версия другая.
   - Причина: pipeline собирает docs с ветки по умолчанию, игнорируя tag/branch release.
   - Вероятность: 5/10.
   - Ущерб: 8/10.
   - Лучшая правка: build docs только по git tag/release ref, сверка `version.json` с tag.

37. binary artifacts size: лимиты на артефакты/страницы
   - Симптом: сборка падает на лимитах CI или хостинга (артефакты слишком большие).
   - Причина: генерация включает крупные бинарники (диаграммы, бандлы) без лимитов.
   - Вероятность: 6/10.
   - Ущерб: 6/10.
   - Лучшая правка: лимит размера артефактов, компрессия, вынос больших файлов в CDN, fail-on-oversize.

38. accessibility regression на generated страницах
   - Симптом: нарушение WCAG (контраст, aria, таб-индексы) после обновления шаблонов.
   - Причина: генератор рендерит элементы без a11y атрибутов.
   - Вероятность: 5/10.
   - Ущерб: 7/10.
   - Лучшая правка: a11y lint в CI, тесты с axe, шаблоны с обязательными aria.

39. API docs drift из-за conditional compilation (Rust)
   - Симптом: docs показывают API, которого нет в сборке пользователя.
   - Причина: генерация без учёта `cfg`/feature flags и target triple.
   - Вероятность: 6/10.
   - Ущерб: 7/10.
   - Лучшая правка: генерировать по ключевым feature-set/targets, явно маркировать условия в docs.

40. CI matrix: разные платформы дают разные docs
   - Симптом: docs отличаются между linux/macos/windows сборками.
   - Причина: toolchain/зависимости ведут себя по-разному, генератор не детерминирован.
   - Вероятность: 4/10.
   - Ущерб: 6/10.
   - Лучшая правка: фиксировать OS для docs, сверка hash между матрицами, deterministic build.

41. license contamination от auto-included sources
   - Симптом: в docs попадают фрагменты, несовместимые с лицензией проекта.
   - Причина: генератор включает внешние файлы/примеры без license-check.
   - Вероятность: 3/10.
   - Ущерб: 8/10.
   - Лучшая правка: allowlist источников, license-scan на входе, запрет неизвестных лицензий.

42. governance/ownership: нет ответственного за generated раздел
   - Симптом: баги в docs висят, потому что нет владельца.
   - Причина: нет DRI и SLA на авто-доку.
   - Вероятность: 5/10.
   - Ущерб: 6/10.
   - Лучшая правка: назначить DRI, ownership-map по папкам, SLO по свежести.

43. incremental migration: «двойная правда» между legacy и новым порталом
   - Симптом: разные правила в старой и новой доке; пользователи получают противоречивые инструкции.
   - Причина: миграция идёт по частям без единого source of truth и редиректов.
   - Вероятность: 6/10.
   - Ущерб: 7/10.
   - Лучшая правка: определить «каноничную» площадку, включить 301 редиректы и frozen-режим для legacy.

44. runtime perf деградирует на большом количестве клиентских страниц
   - Симптом: TTI/INP растут, поиск и навигация «подвисают».
   - Причина: слишком тяжёлые client-side компоненты и большой JSON payload.
   - Вероятность: 5/10.
   - Ущерб: 6/10.
   - Лучшая правка: server-first рендер, lazy-load тяжёлых блоков, пагинация данных, budget на payload.

45. SEO/robots canonical: дубликаты и каноникал указывает не туда
   - Симптом: дубли в индексе, падение позиций.
   - Причина: неправильный `canonical`/`robots` на версиях и локалях.
   - Вероятность: 4/10.
   - Ущерб: 7/10.
   - Лучшая правка: единая политика canonical + hreflang, тесты sitemap/robots, enforce на сборке.

46. multi-branch workflow: веточные доки «затирают» прод
   - Симптом: staging/preview случайно публикуется в production.
   - Причина: отсутствуют изоляция веток и правила публикации.
   - Вероятность: 4/10.
   - Ущерб: 8/10.
   - Лучшая правка: отдельные buckets/paths для веток, publish только из release-тегов, branch protection.

47. supply-chain: компрометированный генератор/плагин
   - Симптом: вредоносный контент или утечка данных через build.
   - Причина: незафиксированные версии, отсутствие signature/lockfile-политики.
   - Вероятность: 3/10.
   - Ущерб: 9/10.
   - Лучшая правка: pin/lock, проверка checksum/SBOM, allowlist пакетов и репозиториий.

48. human review overload: генерируемые PR слишком «шумные»
   - Симптом: review пропускает критические ошибки из-за огромных диффов.
   - Причина: нет дифф-агрегации и разделения ручных/сгенерированных изменений.
   - Вероятность: 6/10.
   - Ущерб: 6/10.
   - Лучшая правка: split commits, отдельные PR для generated, summary-репорт, git diff filters.

49. локализация: неполные/устаревшие переводы живут бесконечно
   - Симптом: часть языков «заморожена» и не обновляется.
   - Причина: отсутствие политики freshness и SLA на локали.
   - Вероятность: 5/10.
   - Ущерб: 5/10.
   - Лучшая правка: метка «needs-translation», fail-fast при просрочке, отчет по coverage.

50. юридические требования на региональные вариации
   - Симптом: в определённых регионах нет обязательных дисклеймеров.
   - Причина: правила/legal контент не ветвятся по юрисдикции.
   - Вероятность: 3/10.
   - Ущерб: 9/10.
   - Лучшая правка: region-aware content rules, обязательные проверки по locale/region.

51. governance: конфликт ownership между docs и core-командой
   - Симптом: правки блокируются, SLA не выполняется.
   - Причина: нет явной границы ответственности за генерацию/ручной слой.
   - Вероятность: 5/10.
   - Ущерб: 6/10.
   - Лучшая правка: RACI/ownership map, single DRI, прописанные SLO и escalation.

52. инкрементальные сборки пропускают зависимые изменения
   - Симптом: обновили базовые правила/шаблоны, а часть страниц не пересобралась.
   - Причина: кеш/инкрементальность без полного dependency graph.
   - Вероятность: 6/10.
   - Ущерб: 7/10.
   - Лучшая правка: dependency graph + invalidation по хэшу шаблонов и схем.

53. VitePress v2 alpha API drift
   - Симптом: после обычного обновления ломаются `data loaders`, хуки или `themeConfig` (без явной причины).
   - Причина: alpha-версия меняется между релизами и не даёт стабильного API-сюрфейса.
   - Вероятность: 6/10.
   - Ущерб: 8/10.
   - Лучшая правка: pin exact version в production, nightly-канал только в отдельный soak-job, contract tests на `route/nav/search`.

54. rustdoc JSON schema drift
   - Симптом: после смены toolchain Rust документация API внезапно частично/иначе сериализуется.
   - Причина: нестабильность nightly-формата rustdoc JSON и схемы.
   - Вероятность: 7/10.
   - Ущерб: 8/10.
   - Лучшая правка: фиксировать `rust-toolchain.toml`, отдельный адаптер schema->внутренний формат и golden-тесты.

55. doctest/snippet drift
   - Симптом: пример в документации компилируется локально раньше, но невалиден для текущего API.
   - Причина: кодовые фрагменты не прогоняются в CI, отсутствует связка doc examples с API evolution.
   - Вероятность: 7/10.
   - Ущерб: 7/10.
   - Лучшая правка: вынос примеров в testable snippets + `cargo test --doc`/отдельный harness в CI.

56. feature/target mismatch в Rust docs
   - Симптом: docs показывают API из другого набора features/target, чем реально у пользователя.

   - Ущерб: 8/10.
   - Лучшая правка: генерировать минимум 2–3 профиля (`default/full/minimal`) и явно маркировать ограничения в frontmatter.

57. namespace collision manual и API routes
   - Симптом: `/rules/foo` и `/api/foo` конфликтуют в поиске/ссылках.

   - Вероятность: 6/10.
   - Ущерб: 7/10.
   - Лучшая правка: жёсткие namespaces (`/rules/**`, `/api/rust/**`, `/api/openapi/**`) + route uniqueness gate.

## 10) Как лучше сделать: итоговые 3 улучшенные модели

### Модель A — GitOps generated + manifest gate

- Уверенность: `9/10`
- Надёжность: `9/10`
- Почему: максимальная предсказуемость, прозрачность review и детект дрейфа через diff.
- Что лучше: подходит, если важен контроль качества и историчность изменений в docs.
- Ключевая команда: `generate -> validate -> git diff --exit-code -> build -> check links -> publish`.

### Модель B — GitOps + fail-fast quality gate

- Уверенность: `10/10`
- Надёжность: `8/10`
- Почему: одинаковая схема как в модели A, но с расширенным gate по 4–7 проверкам.
- Что лучше: для быстрого продления на существующей команде.
- Что добавляется: `onBrokenLinks=error`, check orphan refs по удаленным файлам, route collision check, stale hash check.

### Модель C — Data-first snapshot

- Уверенность: `6/10`
- Надёжность: `10/10`
- Почему: самый устойчивый к evolution API/schema/format изменений и rollback по snapshot.
- Что лучше: если важна долгоживая эволюция форматов и высокий SLA.
- Риск: выше стоимость внедрения и поддержки.

## 12) Альтернативы и гибриды (mdBook, Docusaurus, Starlight, кастом)

Ниже — короткий, практичный срез по альтернативам/гибридам, с акцентом на командный опыт, edge cases и интеграцию авто-доки.

### mdBook (Rust-first)

Где лучше:
- нативно в Rust-экосистеме; легко написать preprocessing на Rust/любом языке через stdin/stdout;
- хорошо для narrative книг, гайдов, примеров, где нужен `{{#include}}` и `{{#rustdoc_include}}`;
- удобная компоновка: строгий pipeline через `book.toml` + preprocessor конфиги.

Где хуже:
- слабее как “портал” со сложной навигацией/версированием/мультисайтом;
- UI/тема и plugin-экосистема беднее в сравнении с Docusaurus/Starlight;
- для сложных API-справочников чаще придётся писать собственный генератор.

Edge cases:
- preprocessor не установлен: сборка падает; нужен строгий контроль зависимостей и проверка окружения.
- include-цепочки и циклы (вложенные `include`): детектить циклы и ограничивать глубину;
- кастомные renderers требуют отдельной настройки и контроля параметров вывода.

### Docusaurus (платформенная зрелость)

Где лучше:
- зрелая система versioning и “заморозки” версий через `docs:version`;
- autogen sidebar из файловой структуры с метаданными.

Где хуже:
- тяжелее в сборке и управлении (React/MDX stack);
- больше “платформенной инерции”, если хочется простую data-driven схему;
- сложнее кастомизировать при большом объёме auto-generated контента.

Edge cases:
- order и коллизии: autogen сортирует по имени, нужны явные metadata для порядка.
- version links: требуют относительных `.md` ссылок, иначе ломаются переходы между версиями.
- рост количества версий резко увеличивает время сборки и размер артефактов.

### Starlight (Astro docs kit)

Где лучше:
- сильный docs UX “из коробки” и быстрый рендер;
- богатая экосистема плагинов и интеграций (OpenAPI, TypeDoc, версии, авто-sidebar и др.).

Где хуже:
- версияция часто через community-плагины, не core-first;
- экосистема менее “enterprise-зрелая” в сравнении с Docusaurus;
- для сложного, многоисточникового портала потребуется доп. слой оркестрации.

Edge cases:
- несовместимость плагинов между версиями Astro/Starlight;
- зависимость от community-плагинов повышает риск поломок при обновлениях;
- при больших док-порталах нужен строгий контроль структуры и ограничений сборки.

### Кастомный pipeline (data-first)

Где лучше:
- максимальный контроль над схемой данных, навигацией, индексами, версиями;
- удобен, если у вас много машинных источников (Rust API, правила, схемы, OpenAPI).

Где хуже:
- самая высокая стоимость разработки и поддержки;
- сложнее onboarding и тестирование (нужно поддерживать собственные генераторы).

Edge cases:
- гарантии целостности (stale/partial generation) должны быть “руками”;
- нужен более строгий CI gate и набор тестов.

## 15) Улучшенная стратегия fresh docs (end-to-end)

Ниже — приоритетный порядок действий, 10–12 пунктов, с явными выгодами и рисками.

1. Определить единый `content root` и зоны: `docs/rules/` (manual), `docs/generated/` (auto) и `.generated/` (build-only). Benefit: чёткое разграничение ownership, меньше конфликтов. Risk: миграция существующих путей потребует времени.
2. Ввести `manifest` слой: `nav-manifest.json`, `sources.json`, `version.json` и `redirects.json` генерируются в `docs/generated/` и публикуются с контентом. Benefit: управляемая навигация и версии. Risk: больше артефактов в репо.
3. Генерация в temp dir и `atomic move` в `docs/generated/`. Benefit: защита от частичных записей. Risk: нужна дисциплина в CI scripts.
4. Валидаторы схемы: `frontmatter` и `manifest` проверяются через schema (JSON Schema/Zod). Benefit: раннее выявление ошибок. Risk: дополнительные false positive на раннем этапе.
5. Gate 1: `stale/drift check` — `source_hash` vs `generated_hash`, иначе fail. Benefit: отсутствие устаревших страниц. Risk: блокирует merge при пропущенной генерации.
6. Gate 2: `route collision + broken links + anchor check` по HTML после build. Benefit: минимизация 404 и мёртвых якорей. Risk: удлинение сборки.
7. Gate 3: `security redaction` и запрет raw HTML в generated MD. Benefit: уменьшение XSS/утечек. Risk: некоторые фичи markdown могут потребовать allowlist.
8. Build-only `search index` на финальной сборке из фактического route map. Benefit: консистентный поиск. Risk: нужен корректный pipeline порядка шагов.
9. Publish модель: versioned path (`/vX/`) + `atomic publish` + глобальная CDN invalidation. Benefit: детерминированный rollout. Risk: дороже инфраструктурно.
10. Rollback: хранить последний N‑snapshot (например 3) и откат через переключение symlink/branch. Benefit: быстрый recovery. Risk: нужен storage и cleanup policy.
11. Observability: метрики `staleness_hours`, `publish_success`, `broken_links_count`, `render_error_rate`, alerting. Benefit: управляемость в проде. Risk: нужно настроить мониторинг.
12. v2 readiness: параллельный build v1+v2 и сравнение артефактов, переход после 2–3 успешных релизов. Benefit: контролируемая миграция. Risk: двойная стоимость сборки на период миграции.

## 15.7) Target operating model (сводка внедрения)

Единая целевая модель для практики внедрения без размытости:

- **Fallback** — автоматический переход на последний успешный snapshot, если текущий релиз не проходит release verification.
- **Rollback** — ручная/управляемая операция после инцидента с уже опубликованным релизом.

### Plane 1 — Control

1. Источник истины:
- `docs/rules/` — ручные policy и процессы.
- входные артефакты генерации (rustdoc/OpenAPI/TypeDoc) — источник API.
- `docs/generated/` — производный слой.

2. Ownership:
- DRI по зонам (`manual`, `generated`, `schema`, `generator`) и policy на смену контрактов.

3. Контроль версий:
- `version.json` фиксирует версии генераторов, toolchain, schema version, шаблоны.

### Plane 2 — Build

4. Генерация только в temp-dir + atomic publish move.
5. Data gates: `schema`, `stale hash`, orphan, slug/route collisions, namespace policy.
6. Build с VitePress после успешного data gate.
7. Render gates: `broken links`, anchor checks, SSR/hydration smoke, search integrity.

### Plane 3 — Release

8. Publish isolation по versioned path (`/vX/`) и atomic alias-switch.
9. Fallback на последний известный хороший snapshot при fail.
10. Rollback по команде с фиксированным runbook и RTO/<5m целевым допуском.

### Plane 4 — Observe

11. Release verification gate после publish:
- smoke canonical routes, версия в опубликованном `version.json`, freshness CDN, search index integrity.
12. Наблюдаемость:
- метрики `publish_success`, `staleness_hours`, `render_error_rate`, `broken_links_count`.

### Уровни зрелости модели

- Base: закрыты Plane 1 + Plane 2 + Plane 3 до шага 8.
- Stable: плюс Plane 3 шаги 9–10 и Plane 4 с базовой мониторинговой связкой.
- Resilient: все 4 Plane + dual-build readiness (v1+v2) и регулярные drills.

## 17) ADR: Comparative Decision Record

### Проблема
Нужна устойчивая автогенерация документации (manual + generated) для rules с гарантией свежести, управляемостью CI/CD и адекватной стоимостью поддержки.

### Требования
- свежесть и предсказуемость (stale/partial generation недопустимы);
- прозрачный review и ownership;
- управляемые риски изменений платформы;
- приемлемая скорость внедрения;
- низкий operational risk;
- поддерживаемость и низкий upgrade‑долг.

### Варианты
1) **VitePress v1.6.4 + контролируемый переход на v2**  
2) **VitePress v2 alpha**  
3) **Data‑first (собственный pipeline, VitePress как viewer или нейтральный renderer)**  

### Критерии оценки
- cost (внедрение и поддержка);
- operational risk (incident risk, rollback, SLA);
- maintainability (сложность изменений, зависимость от экосистемы);
- скорость внедрения;
- управляемость (governance/ownership);
- наблюдаемость (observability);
- устойчивость обновлений (breaking/alpha риск).

### Итоговое решение
**Выбираем вариант 1: VitePress v1.6.4 + контролируемый переход на v2.**  
Причина: лучший баланс стабильности, управляемости и интеграции с Rust‑pipeline при умеренной стоимости поддержки. Это минимизирует операционные риски в ближайших релизах и оставляет путь к v2 после стабилизации пайплайна.

- Confidence: `9/10`
- Reliability: `9/10`

### Why‑not‑others
- **VitePress v2 alpha**: слишком высокий риск обновлений и нестабильности для production, особенно при большом объёме generated‑страниц.  
  - Confidence: `7/10`
  - Reliability: `6/10`
- **Data‑first**: максимальная устойчивость, но дороже и сложнее в разработке и поддержке. Оправдано только при очень жёстких SLA и многолетней эволюции схем.  
  - Confidence: `7/10`
  - Reliability: `10/10`
